# Session ID Formats — Full Archaeology

Complete format survey of Hermes session IDs found in `~/.hermes/state.db` and `~/AppData/Local/hermes/state.db`.

## Current Standard Format

```
YYYYMMDD_HHMMSS_XXXXXX
```

- **Length:** 23 characters
- **Parts:** `timestamp_date` + `_` + `timestamp_time` + `_` + `hex_suffix`
- **Suffix:** 6 hex characters (0-9a-f), random
- **Count in DB:** ~900 sessions
- **Period:** Current (all recent sessions)
- **Example:** `20260619_102733_1edadf`

## Older 8-Hex-Suffix Format

```
YYYYMMDD_HHMMSS_XXXXXXXX
```

- **Length:** 25 characters
- **Suffix:** 8 hex characters
- **Count in DB:** ~50 sessions
- **Period:** Early June (before ~June 15)
- **Example:** `20260601_174914_db332727`
- **Note:** Co-exists with 6-suffix format during the transition period (June 14-15 has both)
- **Impact:** Gets last char clipped by `trunc5(sid, 24)` in `herm` switch dialog

## Cron Task Format

```
cron_<12hex>_YYYYMMDD_HHMMSS
```

- **Parts:** `cron_` prefix + 12 hex chars (pseudo-unique task ID) + `_` + timestamp
- **Count in DB:** ~40 sessions
- **Period:** June 13-18
- **Example:** `cron_1543384cb1da_20260614_084509`
- **Note:** Each cron job run gets its own session ID. The 12-hex portion appears to be a hash of the cron job name/configuration.

## Background Task Format

```
bg_<HHMMSS>_<XXXXXX>
```

- **Example seen:** `bg_120426_cc4ee3`
- **Count:** 1+ (rare)
- **Note:** Likely generated by `hermes --background` or similar background execution mode.

## Short Hex Format (Primordial)

```
XXXXXXXXXXXX
```

- **Length:** 12 hex characters
- **Count in DB:** 12 sessions
- **Probable origin:** Very old Hermes sessions from before the timestamp-based ID format was introduced
- **Example:** `00767e933d69`, `019715c552df`, `045deb488db3`
- **Note:** Cannot be derived from timestamp — these are purely random

## UUID Format

```
XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX
```

- **Length:** 36 characters
- **Count in DB:** 11 sessions
- **Probable origin:** Migration artifacts from an older Hermes version or cross-instance handoff
- **Example:** `1e8eb132-dd36-4357-b8f2-afc40dcf1d81`
- **Note:** Standard RFC 4122 UUID

## Detection Queries

All queries run against `state.db`:

```sql
-- Current standard (6 hex suffix, 23 chars)
SELECT COUNT(*) FROM sessions WHERE id GLOB '??????_??????_??????';

-- 8 hex suffix (25 chars)
SELECT COUNT(*) FROM sessions WHERE id GLOB '??????_??????_????????';

-- UUID format (with hyphens)
SELECT COUNT(*) FROM sessions WHERE id LIKE '________-____-____-____-____________';

-- 12-hex short format
SELECT COUNT(*) FROM sessions WHERE id GLOB '????????????' AND id NOT LIKE '%\_%' ESCAPE '\';

-- Cron format
SELECT COUNT(*) FROM sessions WHERE id LIKE 'cron\_%' ESCAPE '\';

-- Background format
SELECT COUNT(*) FROM sessions WHERE id LIKE 'bg\_%' ESCAPE '\';

-- All non-standard formats combined
SELECT id FROM sessions WHERE id NOT GLOB '??????_??????_*';
```

## Suffix Resolution

When a user gives a hex fragment that they recall from a session:

| Fragment length | How to search |
|----------------|---------------|
| 4 hex chars | `LIKE '%<fragment>'` — may match multiple sessions |
| 6 hex chars | `LIKE '%<fragment>'` — likely unique, matches both 6-suffix and as substring of 8-suffix |
| 8 hex chars | `LIKE '%<fragment>'` — matches 8-suffix IDs directly. Also try first/last 6 chars. |
| 12 hex chars | = the full short-hex ID, search by exact match |

## Compression Chain Example

A typical compaction chain showing parent->child->tip relationships:

```
20260614_161550_5f4658 ──→ 20260614_161922_ddade7 ──→ 20260614_162124_a46a59 ──→ 20260614_162318_520ff8 ──→ ...
```

Each step has `end_reason='compression'` on the parent and `parent_session_id` set on the child pointing to the parent. The `roots()` function walks this chain to find the latest active session.

## Key Takeaway

The Hermes session DB is NOT single-format. Sessions from different time periods, sources, and execution modes use different ID schemes. Always search across formats when a direct ID lookup fails.
